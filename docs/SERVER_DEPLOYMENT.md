# KQ Remote Link Server Deployment

KQ Remote Link keeps RustDesk's remote-control protocol, so production use
should run a private RustDesk ID server (`hbbs`) and relay server (`hbbr`).

## Ports

Open these ports on the server firewall/security group:

- TCP `21115`: TCP NAT type test
- TCP `21116`: ID/rendezvous service
- UDP `21116`: UDP hole punching
- TCP `21117`: relay service
- TCP `21118`: web client support, if used later
- TCP `21119`: web client support, if used later

If you need a handoff document for cloud/security-group approval, generate one
with:

```powershell
.\scripts\new-kq-server-port-request.ps1 `
  -ServerHost "remote.example.com" `
  -PublicIp "x.x.x.x" `
  -Requester "your-name" `
  -Environment "test"
```

Minimum request: `21115/tcp`, `21116/tcp`, `21116/udp`, `21117/tcp`.
Recommended request: `21115-21119/tcp` and `21116/udp`.

## Docker Compose

From the `deploy` directory on a Linux server:

```powershell
docker compose -f rustdesk-server.compose.yml up -d
```

Or use the deployment helper:

```bash
cd deploy
INSTALL_DIR=/opt/kq-remote-link-server ./deploy-rustdesk-server.sh
```

After deployment, run the health check helper on the server:

```bash
INSTALL_DIR=/opt/kq-remote-link-server ./check-rustdesk-server.sh
```

For Gitea Actions deployment, see `docs/GITEA_SERVER_DEPLOYMENT.md`. The
standard Gitea path for this repository is:

- Workflow: `.gitea/workflows/deploy.yml`
- Deploy script: `scripts/deploy/deploy.sh`
- Deploy directory: `/www/wwwroot/KQromoteLink`
- Public host: `43.154.197.96`

This compose file uses Docker host networking, matching RustDesk's OSS server
deployment path for Linux. `hbbs` and `hbbr` run with `-k _`, so RustDesk
manages a persistent server key pair instead of running without a usable
private-server key. The server key pair is stored under `deploy/data`.
After the first start, copy the public key from:

```text
/www/wwwroot/KQromoteLink/data/id_ed25519.pub
```

Put that value into `deploy/custom-client.example.json` as `key`.

If the key file is missing after containers are already running, export it from
the container or `hbbs` logs:

```bash
INSTALL_DIR=/www/wwwroot/KQromoteLink ./deploy/export-hbbs-public-key.sh
```

The deploy and health-check scripts also run the same export logic
automatically and fail if no public key can be found.

For repeatable client packaging, generate the hbbs/hbbr key pair before
deployment:

```powershell
.\scripts\new-kq-server-key-pair.ps1
```

Keep the generated secret key outside git. Provide the two values to the Gitea
runner environment or repository secrets as:

```bash
KQ_HBBS_PUBLIC_KEY=<generated public key>
KQ_HBBS_SECRET_KEY=<generated secret key>
```

When both variables are present, `deploy/deploy-rustdesk-server.sh` writes them
to `/www/wwwroot/KQromoteLink/data/id_ed25519.pub` and
`/www/wwwroot/KQromoteLink/data/id_ed25519` before starting containers.

The deploy script also tries to open local firewall rules for
`tcp/21115-21119` and `udp/21116`. Cloud security groups still need to allow the
same ports.

## Client Configuration

Replace these placeholders in `deploy/custom-client.example.json`:

- `43.154.197.96:21116`: your `hbbs` host and port
- `43.154.197.96:21117`: your `hbbr` host and port
- `api-server`: your API/server URL if you expose one; leave blank if not used
- `PASTE_HBBS_PUBLIC_KEY_HERE`: public key generated by `hbbs`

RustDesk reads signed custom-client settings from `custom.txt`. This branch adds
a standalone `tools/custom_client_signer` utility so you can generate your own
signing key and sign the JSON template without pulling in the full RustDesk app
package.

Private server fields are written to both `default-settings` and
`override-settings`. This is intentional: `override-settings` wins over any
existing user config such as a previously saved public RustDesk server, so
upgrades and test machines keep using the packaged private hbbs/hbbr.

Generate a key pair:

```powershell
.\scripts\new-kq-custom-client-config.ps1 -GenerateKeyPair
```

The command prints:

```text
public-key=<base64 public key>
secret-key=<base64 secret key>
```

Build the app with the public key:

```powershell
$env:KQ_CUSTOM_CLIENT_PUBKEY = "<base64 public key>"
.\scripts\build-windows-flutter.ps1 -SkipPortablePack
```

Sign the client config with the secret key:

```powershell
.\scripts\new-kq-custom-client-config.ps1 `
  -RendezvousServer "43.154.197.96:21116" `
  -RelayServer "43.154.197.96:21117" `
  -ServerKey "<hbbs public key>" `
  -SecretKey "<base64 secret key>" `
  -PublicKey "<base64 public key>" `
  -CopyToReleaseDir ".\flutter\build\windows\x64\runner\Release"
```

Place the generated `custom.txt` next to the app executable or package it with
your release artifact. The relevant runtime keys are:

- `app-name`
- `custom-rendezvous-server`
- `relay-server`
- `api-server`
- `key`
- `hide-server-settings`

For the normal production packaging path, use the wrapper script so the signed
`custom.txt` is generated, verified, copied into the zip, and checked by the
release smoke test:

```powershell
.\scripts\new-kq-private-server-client-package.ps1 `
  -RendezvousServer "43.154.197.96:21116" `
  -RelayServer "43.154.197.96:21117" `
  -ServerKey "<hbbs public key>" `
  -PublicKey "<base64 custom-client public key>" `
  -SecretKey "<base64 custom-client secret key>"
```

If you want the script to create a fresh custom-client signing key pair, add
`-GenerateSigningKey`. If the current Windows release was not already built
with the matching public key, add `-BuildClient` so it rebuilds with
`KQ_CUSTOM_CLIENT_PUBKEY` before packaging.

The script refuses to package by default unless `-BuildClient` is supplied,
because `custom.txt` is accepted only by an executable built with the matching
custom-client public key. If you intentionally package an existing release that
was already built with the same key, pass `-UseExistingBuildWithMatchingKey`.

If you already have a `custom.txt`, verify that it matches the public key baked
into the client:

```powershell
cargo run --manifest-path tools/custom_client_signer/Cargo.toml -- verify custom.txt "<base64 public key>"
```

Before distributing clients, check DNS and the RustDesk server ports from a
client-side network:

```powershell
.\scripts\test-kq-server.ps1 `
  -RendezvousServer "43.154.197.96:21116" `
  -RelayServer "43.154.197.96:21117"
```

After operations returns `/www/wwwroot/KQromoteLink/data/id_ed25519.pub`, pass
it as `-ServerKey` to validate the key format before building a private-client
package.

Or read the values from a generated custom-client JSON:

```powershell
.\scripts\test-kq-server.ps1 -ConfigJson .\deploy\custom-client.generated.json
```

## Smoke Test

1. Start `hbbs` and `hbbr`.
2. Build and launch two KQ Remote Link clients with the same server config.
3. Confirm both clients show an ID and are online.
4. From client A, connect to client B by ID.
5. Verify password/accept flow, remote screen, keyboard/mouse, clipboard, file
   transfer, and relay fallback by testing from different networks.
